# План A/B-тестирования моделей

## Концепция

**Цель** A/B-теста - сравнить две версии модели прогнозирования дефолта по кредитным картам и определить, даёт ли новая модель улучшение по качеству выявления клиентов с высоким риском дефолта.

**В тесте сравниваются**:

* **model_v1** - базовая модель
* **model_v2** - новая версия модели

Пользователи или входящие заявки случайным образом делятся на две группы с распределением трафика 50/50. Такое разделение позволяет сравнить модели в одинаковых условиях.

* **Группа A** - получает прогноз от `model_v1`;
* **Группа B** - получает прогноз от `model_v2`.

**Рекомендуемая продолжительность A/B-теста**: 2-4 недели

Тест должен длиться достаточно долго, чтобы накопить репрезентативное количество заявок и снизить влияние случайных колебаний


## ML-Метрики

***Основная метрика:*** F1-score для класса дефолта

**F1-score** выбран, потому что в задаче кредитного скоринга важно учитывать одновременно:

**Precision** - насколько точно модель определяет клиентов с риском дефолта

**Recall** - какую долю реальных дефолтов модель смогла обнаружить


***Дополнительные метрики:***

* Precision
* Recall
* ROC-AUC

Precision - показывает, какая доля клиентов, предсказанных как дефолтные, действительно оказалась дефолтной. Высокий Precision снижает количество ложных отказов надёжным клиентам.

Recall - показывает, какую долю реальных дефолтов модель смогла найти. Высокий Recall снижает риск пропустить клиента, который может не вернуть кредит.

ROC-AUC - показывает общую способность модели разделять клиентов на дефолтных и недефолтных.

## Бизнес-метрики

Для банка важны не только ML-метрики, но и бизнес-эффект.

В качестве бизнес-метрик можно использовать:

* ***Ожидаемые финансовые потери***

Expected Loss = PD × LGD × EAD

где:

* PD - вероятность дефолта, предсказанная моделью
* LGD - доля потерь при дефолте
* EAD - сумма задолженности на момент дефолта

Цель - снизить ожидаемые финансовые потери без чрезмерного сокращения числа одобренных заявок.

* ***Доля одобренных заявок при допустимом уровне риска***

Эта метрика показывает, сколько клиентов может быть одобрено при сохранении приемлемого уровня дефолта.

Цель - не только снижать риск, но и не отклонять слишком много потенциально хороших клиентов.

## Статистический анализ

Для сравнения моделей используется статистическая проверка различий между группами A и B.



* Для долевых метрик, например Precision или Recall, целесообразно использовать **Z-test для разности долей**

* Для средних бизнес-метрик, например ожидаемых финансовых потерь, будет использовать **T-test Стьюдента для независимых выборок**

* Для ключевых метрик строятся **95% доверительные интервалы**

## Критерий успешности теста

Новая модель model_v2 считается успешной, если выполняются следующие условия:

1. F1-score для класса дефолта выше, чем у model_v1
2. улучшение статистически значимо при уровне значимости 0.05
3. Recall не снижается относительно model_v1
4. бизнес-метрики не ухудшаются
5. доля ложных отказов надёжным клиентам остаётся на приемлемом уровне

## Логирование результатов

Для последующего анализа необходимо сохранять:

* версию модели
* входные признаки клиента
* предсказанный класс
* вероятность дефолта
* фактический результат после наблюдаемого периода
* timestamp запроса
* группу A/B-теста

В текущем проекте API уже логирует предсказания в JSON-формате. В production-среде такие логи могут передаваться в систему мониторинга и аналитики, например ELK-стек.

## Ограничения

В рамках учебного проекта A/B-тест описан концептуально

Фактическая онлайн-проверка невозможна без реального потока заявок и последующего наблюдения факта дефолта
согласованного порога вероятности дефолта.

Поэтому в проекте реализована техническая возможность выбора версии модели, а сам A/B-тест представлен как план промышленного эксперимента.
