# Credit Card Default Prediction Service

Проект выполнен в рамках дисциплины «Внедрение моделей машинного обучения».

Цель проекта - разработка и внедрение сервиса прогнозирования дефолта по кредитным картам на основе исторических данных клиентов. В проекте реализован полный цикл вывода модели машинного обучения в production-подобную среду: подготовка данных, обучение моделей, создание REST API, разработка пользовательского интерфейса, логирование, тестирование и контейнеризация.

В качестве источника данных используется набор UCI Credit Card Default Dataset.

Тип задачи: **Бинарная классификация**

Целевая переменная: **target**

Значения:

* **0** - клиент не допустил дефолт
* **1** - клиент допустил дефолт

## Используемые модели

В проекте реализованы две версии модели.

* Model V1: Logistic Regression - Используется как базовая модель.
* Model V2: Random Forest Classifier - Используется как альтернативная модель для сравнения качества и проведения A/B-тестирования.

## Сервисная архитектура


В рамках данного проекта был выбран монолитный подход.

Архитектура состоит из одного Flask-приложения, которое объединяет:

* REST API
*	веб-интерфейс пользователя
*	загрузку моделей машинного обучения
*	выполнение предсказаний
*	логирование запросов и ответов

## Схема взаимодействия компонентов:
```text
Пользователь => Flask API + GUI => Model Handler => ─── Model V1
                                                    └── Model V2
```
Монолитная архитектура была выбрана по следующим причинам:

1.	небольшой размер проекта
2.	единый источник данных
3.	отсутствие необходимости независимого масштабирования компонентов
4.	упрощение развертывания и сопровождения приложения
5.	сокращение количества инфраструктурных зависимостей

В промышленной среде возможен переход к микросервисной архитектуре с выделением отдельных сервисов для API, инференса моделей и мониторинга.

## Концепт использования брокера сообщений
В текущем проекте все запросы обрабатываются синхронно через HTTP API.
При росте нагрузки возможно использование брокера сообщений RabbitMQ.

Потенциальные сценарии применения:

*	пакетная обработка заявок клиентов
*	асинхронный расчёт прогнозов
*	обработка больших исторических выборок
*	доставка событий логирования
*	интеграция с внешними банковскими системами

Возможная схема работы:
```text
Клиент => REST API => RabbitMQ => Worker => ML Model
```
Такой подход позволяет отделить приём запросов от вычислений и повысить масштабируемость системы.

## Логирование и мониторинг
В проекте реализовано структурированное логирование событий API.
Логи сохраняются в файл: logs/api_logs.jsonl

Каждая строка представляет собой отдельный JSON-документ.

Пример записи:
```bash
{"timestamp": "2026-06-13T21:35:58.101425+00:00",
  "event_type": "prediction_success",
  "source": "api",
  "model_version": "v1",
  "prediction": 1,
  "probability": 0.8091}
```
Логируются:

*	успешные запросы
*	ошибки валидации
*	ошибки загрузки моделей
*	ошибки обработки запросов
*	
В production-среде такие логи могут собираться средствами ELK Stack:

* Elasticsearch
* Logstash
* Kibana

Это позволяет строить дашборды мониторинга и анализировать работу моделей.

## Инструменты MLOps

Фактическая интеграция MLOps инструментов в проект не выполнялась, но рассматривается как возможное направление дальнейшего развития.

**DVC** (Data Version Control) предназначен для контроля версий данных и артефактов машинного обучения.
В данном проекте DVC мог бы использоваться для:

*	хранения различных версий датасета
*	версионирования обученных моделей
*	обеспечения воспроизводимости экспериментов

**MLflow** используется для отслеживания экспериментов машинного обучения.

В рамках данного проекта MLflow мог бы применяться для:

*	хранения метрик моделей
*	сравнения различных алгоритмов
*	отслеживания гиперпараметров
*	регистрации версий моделей перед развёртыванием

## A/B-тестирование
В проекте предусмотрена возможность сравнения двух моделей:

*	model_v1;
*	model_v2.

Детальный план проведения эксперимента представлен в файле: **AB_TEST_PLAN.md**

В рамках проекта реализована техническая возможность выбора версии модели через API и GUI, что позволяет использовать сервис как основу для дальнейшего внедрения полноценного A/B-тестирования.
